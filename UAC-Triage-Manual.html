Linux UAC Capture Analyzer - Field Manual · covers v1.8.0 · triage a Unix-like Artifacts Collector (UAC) capture on Windows

A single-file Windows HTA that turns a UAC collection from a Linux / macOS / *BSD / Solaris / ESXi host into scored, searchable triage views - offline, with no dependencies beyond Windows' built-in tar.exe, PowerShell, and (optionally) WSL. GitHub: bpmorris22/Linux-UAC-Capture-Analyzer.

What changed since the v0.9 manual. The tool was renamed UAC Triage Tool → Linux UAC Capture Analyzer at v1.0 (the file is now Linux-UAC-Analyzer.hta; the _Processed\<host>\UAC output convention and SQLECmd sidecar filenames are unchanged for continuity). Whole new tabs have arrived: Security - an automated assessment scored PASS / WARN / FAIL / N-A (attack-chain correlations plus a CIS/NIST-inspired OS and Apache hardening posture); Users, SSH Keys, Actions (the full executed-command record, scored across the attacker playbook), Apache Web Server (config summary + access logs - renamed from "Apache logs"), Audit, and dedicated Apps / Persistence tabs (graduated off the Overview). Cross-cutting: a Non-Business Hours filter on every time-based tab, a raised display cap with a prominent Maximum Rows Exceeded warning, per-tab headings/descriptions, resizable+frozen columns, online geo/ASN enrichment, and over-time charts on Logons and Network. Screenshots below are from the early builds (synthetic demo data) - layouts have gained columns and tabs since, but the anatomy they show still applies. New in v1.7: the Persistence tab gained RAT / C2 sweeps - system-wide login scripts, SSH-login hooks (sshrc, ~/.ssh/rc), run-parts cron dirs, network-event triggers, module-autoload, udev RUN, binfmt, at jobs - plus a C2 tradecraft lens on every command line (reverse-shell/backpipe/encoded-loader patterns, a known-implant name watchlist, and hardcoded callback IPs). v1.7.1 deepened this: the content-sweep gate now surfaces implant-by-name and reverse-shell lines that carry no obvious token; the lens gained php/ruby/gawk and /dev/udp reverse shells, a beacon-cadence rule, and IPv6/domain callbacks; and new vectors are swept (systemd Environment=LD_PRELOAD / socket ListenStream= / generators, ld.so.conf.d, authorized_keys environment=). v1.8.0 added the structural piece a second review asked for: a persistence entry's executable is followed to a running process and its sockets, so a harmless-looking service becomes high-confidence when its binary is running and holding a public connection (PERS-C2). It also parses systemd drop-in overrides (<unit>.service.d/*.conf), the SSH auth-source config (AuthorizedKeysCommand, TrustedUserCAKeys, PermitUserEnvironment), the PAM account/password/session stacks, cloud-init per-boot scripts and D-Bus activation, adds tunnel detection (autossh, ssh -R, cloudflared/ngrok), and fixes the long-broken ForceCommand check plus the over-broad implant-name match. New in v1.9: a dedicated Containers tab (left of Apache Web Server; the container table moved off Apps) scoped to the evidence the capture actually holds - a collection-coverage map, docker/podman runtime metadata with findings, an inventory that keeps stopped containers visible, a per-container inspect deep-dive (lifecycle, execution config, redacted environment, isolation and exposure scoring incl. docker.sock/capabilities/unconfined, mounts and network with host corroboration), scored docker diff filesystem changes, and container-to-host correlations (inspect State.Pid to the process list, docker top host PIDs, Timeline and Events pivots). v0.8–v0.9 recap: browser-DB detection + SQLECmd hand-off and the capture-profile pill & recapture warning.

1. What it is

UAC (Unix-like Artifacts Collector) is excellent at collecting a Unix-like host: it produces a uac-<host>-<os>-<timestamp>.tar.gz (or .zip) containing plain-text command output under live_response/, a mirrored filesystem under a directory literally named [root], an optional TSK bodyfile, and hash sets. This tool is about reading that result fast on a Windows analysis box - surfacing the handful of rows that matter out of a capture with tens of thousands of files.

  • It never runs UAC and never touches the target. No SSH, no remote anything. The archive is the entire interface.
  • Everything is scored. Additive heuristic rules per row; ≥3 is flagged suspicious, ≥5 is hot. An IOC/keyword list sharpens everything.
  • Every finding carries its MITRE ATT&CK technique, and the Overview summarizes the distinct techniques observed.
  • It degrades honestly. Logs-only captures, non-root captures, and long-path extractions all state exactly what is and isn't covered instead of showing silently empty views.

2. Quick start

  1. On the *nix host, download UAC from tclahr/uac releases, then run it from its own folder: 
    cd uac-3.3.0
./uac -p ir_triage /tmp          # full live-response triage (recommended)
./uac -a files/logs/* /tmp       # logs-only (lighter; Logons/Events/Logs carry the signal)
  2. Copy the resulting archive to the Windows analysis box.
  3. Open UAC-Triage-Tool.hta, point it at the archive (or an already-extracted folder), confirm the Target hostname, and click Analyze capture.
  4. Read Overview first (metadata, caveats, top findings, ATT&CK summary), then work Processes → Network → Logons → Events → Logs. Anything scored ≥3 deserves an explanation.
Output directory: keep it short (e.g. C:\Cases\<host>). UAC captures nest ~170 characters deep below the extract root; a deep output directory pushes files past Windows' classic 260-character limit and out of the parsers' reach (§4, §19).

3. Command line

mshta "UAC-Triage-Tool.hta" "<archiveOrFolder>" ["<outDir>"] [/auto] [/from:yyyy-MM-dd] [/to:yyyy-MM-dd]
ArgumentMeaning
<archiveOrFolder>A uac-*.tar.gz / .tar / .zip, or a folder that already contains uac.log.
<outDir>Where extraction + runinfo land. Defaults to _Processed\<host>\UAC next to the app.
/autoExtract and parse immediately, no click needed.
/from: /to:Case window (UTC). Prefilters the Timeline, Logons, Events and package activity. Filter only - never affects scoring.

A shared IOC.txt next to the app is auto-merged into the IOC box at launch.

4. The capture panel

Extraction uses Windows' built-in tar.exe (bsdtar), which transparently handles .tar/.tar.gz/.zip. Two behaviours are normal and expected:

  • bsdtar exits 1 on every UAC tarball - the archive carries Unix symlinks Windows can't create. The regular files all extract; success is judged by uac.log appearing, never by the exit code.
  • Encrypted zips (UAC -P) can't be opened by bsdtar - extract with 7-Zip first and use Pick folder….

Capture profile pill & recapture warning v0.9

After Analyze, a color-coded pill in the header shows the collection profile read from uac.log's command line (-p/--profile, or a bare -a artifact list):

  • FULL - green: the complete artifact set; nothing to warn about.
  • IR TRIAGE / OFFLINE IR / CUSTOM - amber: a subset; user-content artifacts (browser databases, mail, the wider [root] mirror) and/or live response are excluded.
  • OFFLINE - blue: disk-only against a mounted image; live response is impossible for this capture type, so it is informational rather than a nag.
  • LOGS-ONLY - red: the most restricted list; Processes/Network/Timeline are empty by design.
  • PROFILE ? - grey: no profile switch found on the command line.

Click the pill at any time for the coverage breakdown. When a capture is positively a lesser live profile (ir_triage, offline_ir_triage, logs-only, custom), a warning popup opens automatically once per Analyze: it lists the artifact groups that profile excludes (indicative, not a YAML diff), calls out non-root collection gaps, and offers a copyable recapture command that keeps the original -f zip flag and destination, e.g.:

sudo ./uac -p full /tmp

Run it from the UAC folder on the target if the host is still available - a full capture is larger and slower than ir_triage, but it is the only way to get the excluded user-content artifacts (including the browser databases the SQLECmd hand-off needs). Offline and unknown-profile captures never trigger the popup.

Long paths v0.5.1

systemd journal filenames alone are ~92 characters, and captures nest ~170 characters below the extract root. bsdtar happily creates paths past Windows' classic 260-character MAX_PATH, but the HTA's file layer cannot read them - so the app:

  • pre-flights the output directory length and warns before extraction if the tree would exceed the limit;
  • lists the extracted tree via robocopy (long-path capable - cmd dir /s /b is not) and audits every path >259 chars, reporting the count on the Overview and in Logs → Log integrity;
  • computes journal coverage from filenames via that listing, so it survives at any depth.

Rotated .gz logs v0.4

Rotated logs (auth.log.2.gz, syslog.3.gz, …) are inflated at parse time via PowerShell's GzipStream (idempotent - existing inflated siblings are skipped). Weeks of rotated auth/syslog/dpkg/apt history are parsed automatically; on a logs-only capture that's usually where the real activity lives.

5. Overview tab

v0.3 screenshot (synthetic demo data). The current Overview adds ATT&CK coverage, integrity/journal-gap/config-drift stat cards, coverage caveats, a When column on findings, and package attribution.

Run metadata & coverage caveats

Parsed from uac.log plus uptime/date: host, OS/arch, UAC version, collection time and host time zone, boot time, uptime/load, the exact command line, and failed collector commands. Three caveat lines matter:

  • Collection profile - a logs-only run (-a files/logs/*) is detected from the command line; Processes/Network/Timeline then explain themselves as "not collected in this profile" instead of looking broken.
  • Non-root collection - when Running as isn't root, the Overview states what's typically missing (shadow, sudoers, other users' /proc detail, btmp) and summarizes permission-denied paths (capped - a 28 MB uac.log produced 1,479 of them).
  • Long paths - count of files beyond the 260-char limit that the parsers cannot read (§4).

Datasets, Top findings & ATT&CK summary v0.6

Stat cards per dataset (flagged counts highlighted), a cross-dataset Top findings list (highest scores first, including integrity findings and suspicious packages), and a row of ATT&CK technique chips - the distinct techniques observed across all flagged rows (hover a chip for the technique name).

Persistence & system findings expanded v0.5–v0.7

Scored command lines and findings from: /etc/crontab, cron.d, per-user crontabs, systemd unit ExecStart= lines and live systemd timers, rc.local, ld.so.preload (non-empty = +3), authorized_keys forced commands, account findings (extra UID-0 accounts, service accounts with shells, docker-group members), privilege sweeps (suid outside standard paths, dangerous file capabilities, world-writable system files, hidden files in temp/system locations - stock entries whitelisted), UAC's null-passphrase SSH private keys, shell/python/vim/SQL history one-liners (with timestamps when HISTTIMEFORMAT was set - shown in the When column), and config drift (§14). Benign lines that scored 0 are counted but hidden.

Browser databases → SQLECmd v0.8

A sweep detects browser SQLite databases collected under user profiles - Firefox places.sqlite / cookies.sqlite / formhistory.sqlite in the classic, snap and flatpak layouts, and Chromium-family History / Cookies / Web Data / Login Data (Chrome, Chromium, Edge, Brave, Vivaldi, Opera, incl. snap). When any are found, this section lists them (browser, user, size, -wal sidecar presence) and the Datasets column gains a browser DBs card. Detection is long-path safe (robocopy listing first). v0.9.2 The section sits directly below Persistence & system findings, the browser DBs stat card jumps straight to it when clicked, and Analyze all in SQLECmd is the blue primary button.

  • SQLECmd → per row launches the family SQLECmd wrapper with that database prefilled; Analyze all passes the common parent folder as directory input so every database is map-matched in one run. SQLite is platform-agnostic - the Zimmerman maps parse Linux-collected Chrome/Firefox DBs unchanged.
  • Output goes to _Processed\<host>\SQLECmd beside this app's own \UAC output - wrapper-specific folders, no collisions. The case window (/from /to) is forwarded. Nothing runs automatically: review the wrapper's input and click Process there.
  • Wrapper location: SQLECmd-Wrapper.hta is found automatically in this app's folder or v0.9.3 in any sibling tool folder (the family layout: …\Artifacts\<Tool>\ side by side).
  • Auto-download / update v0.9.4 - if the wrapper isn't found anywhere, a hand-off offers to download the latest SQLECmd-Wrapper.hta from GitHub straight into this app's folder (no hunting for it; browsing to an existing copy stays as the offline fallback). If the copy it finds is older than the latest release, the hand-off offers to fetch the newer one first - Cancel just uses the copy on disk. The Browser-databases hint shows the resolved wrapper's version and whether a newer one is available.
  • WAL caveat: an absent -wal sidecar on a host where the browser was running at collection means the newest visits may not be in the main DB - recollect including -wal/-shm if recent history matters.
  • Profile caveat: the default ir_triage profile does not collect browser databases (ground truth: a snap Firefox came back with only a Crash Reports log). Collect with the full profile or add -a files/browsers/* when web activity matters - otherwise this section simply stays hidden.

Recent package activity v0.4

Preferred source is apt/history.log (+rotations): every install/upgrade/remove/purge with attribution (Requested-By or the commandline) - separating human apt install from unattended-upgrade noise - plus a suspicious-tooling watchlist (nmap, netcat, socat, masscan, hydra, xmrig, sqlmap, …) that tints matching rows. Falls back to dpkg.log / yum.log.

6. Security tab v1.6

The Security tab is an automated assessment computed over everything the app already parsed - it reads no new files of its own (bar a couple of cheap direct reads like sudoers), and it is recomputed fresh on every view, so online geo/ASN enrichment and a re-applied IOC list are always reflected. Every check reads one of four verdicts - FAIL, WARN, PASS, or N-A - and a banner at the top rolls up the counts. N-A means the source artifact was not collected (e.g. no /etc/shadow on a non-root capture, no process list on a logs-only one), so a partial capture never shows a false all-clear. Findings are heuristic leads to confirm, not verdicts: click any row to pivot the Timeline to its evidence or jump to the source tab, and use Export → CSV for the whole set. Three tables:

Compromise indicators & attack-chain correlations

Cross-dataset joins that no single tab can show - the picture emerges only when two artifacts line up:

  • Brute-force that succeeded - a source IP with ≥5 failures then an accepted login.
  • Interactive login from a public IP, and login outside business hours (see the Non-Business Hours definition).
  • Offensive tooling / reverse shell run - reverse-shell, scanner, web-shell-drop or attack-tool commands in shell history.
  • Download → make-executable chain - the same user both fetched a file over the network and chmod +x'd one (the staged-payload sequence).
  • Anti-forensics / log tampering, and an account created or modified from a shell.
  • Persistence changed after a suspicious logon - cron / systemd / authorized_keys / rc.local mtimes (from the bodyfile) that fall after the first suspicious logon.
  • Data staging / exfiltration (remote copy, or an archive staged under a temp path), an SSH key authorized on multiple accounts, a null-passphrase private key, timestomp candidates (crtime > mtime), and logins from multiple countries.

OS hardening posture (CIS/NIST-inspired)

A configuration and account checklist: direct root SSH, password / empty-password auth and extra sshd directives (X11Forwarding, MaxAuthTries, PermitUserEnvironment), PAM auth-stack weaknesses, extra UID-0 and duplicate-UID accounts, blank passwords and password-aging (PASS_MAX_DAYS), service accounts with a login shell, sudoers NOPASSWD grants, privileged / host-namespace containers, auditd presence, the firewall on/off state (ufw / ufw.conf / firewalld) and its ruleset vs 0.0.0.0 listeners, non-standard listening services on externally-reachable interfaces (the "why is this agent on port 8000" check), SELinux/AppArmor, kernel sysctls (ASLR, fs.suid_dumpable, SYN cookies, ip_forward judged against whether containers are present), /tmp mounted noexec,nosuid, NTP clock sync (also an evidence-integrity signal), patch recency, EOL distro, and docker/sudo/wheel membership.

Apache hardening posture (CIS Apache HTTP Server 2.4)

Present only when the host runs Apache. Over the enabled configuration (see the Apache Web Server tab): ServerTokens / ServerSignature, TraceEnable, run-as user (FAIL if root), directory listing (Options Indexes), mod_status exposure, attack-surface modules (WebDAV / userdir / info / CGI / autoindex / SSI), TLS presence and deprecated protocols, AllowOverride All, and access/error logging.

Tree view v0.7

The Tree view button swaps the flat table for a collapsible parent/child hierarchy: fold subtrees with ▸/▾, click any node for its detail pane. Search and flagged only hide non-matching leaves but keep the path to a match visible, so you always see where a flagged process sits (e.g. sshd → bash → curl).

Detail pane

Click any row: full fields, scoring reasons with ATT&CK techniques, Ancestry (parent chain to init) and Children, CPU/Mem at collection, SSH origin (from SSH_CONNECTION in environ, when readable), open sockets and deleted-files-held-open from lsof. Copy produces a case-note-ready block.

Field-tuned noise control: GNOME/Wayland/PipeWire named memfds, renamed runtime threads (MainThread, tokio workers), cache/fontconfig deleted fds, and libraries upgraded under a running process are all recognized and either excluded or scored low with the benign explanation stated - verified against a real desktop Linux capture.

8. Network tab

v0.3 screenshot. The current build adds /proc/net cross-checking and non-root ownership recovery.

From ss -tanp/-uanp (fallback netstat -anp), each socket linked to its owning process. Two v0.5 mechanisms matter on real captures:

  • Hidden-socket cross-check: /proc/net/tcp|tcp6|udp|udp6 are hex-decoded and compared against ss. A LISTEN socket present in the kernel table but missing from ss is flagged +3 - the rootkit-hidden-listener tell. (LISTEN only: ESTABLISHED rows churn between the two commands and would false-positive.)
  • Non-root attribution recovery: a non-root ss shows no process column at all. Ownership is recovered from /proc/net's uid (→ passwd) and per-PID fd socket inodes (→ pid/name), so most sockets still say who owns them.

9. Logons tab

v0.3 screenshot. The current build adds User/Source/Result filters, raw-file links, and many more event types.

Sources: auth.log/secure and all rotated siblings - parsing both ISO-8601 (Ubuntu 22.10+/Debian 12+ default) and classic syslog timestamps - plus wtmp / btmp / lastlog decoded directly from their binaries (essential for logs-only captures), and last -a -F/lastb text (deduplicated: text is only used when the matching binary produced nothing; last -a -F is preferred because it prints the full year).

Event types: SSH accepted/failed (with key fingerprints), sudo (with the command), pkexec (polkit privilege escalation - often missed), pam sessions (root-cron noise is tallied and hidden, not listed), account changes (useradd/usermod/passwd/group*), su, per-account lastlog.

  • Filters v0.7 - User / Source-artifact / Result dropdowns, combinable with search, flagged only and the case window.
  • Raw-file links v0.7 - the Source column links straight to the underlying file (auth.log, wtmp, btmp, lastlog, last output) in Explorer for manual verification.

10. Timeline tab

v0.3 screenshot; anatomy unchanged (Size/UID now center-aligned).

The TSK bodyfile - full filesystem MAC-times. A full-profile bodyfile is MFT-scale (150k+ rows on a minimal box), so the stored/scored set is capped at 50,000 rows with the true total reported; set the case window to focus a big capture (the prefilter runs at parse time, so a window makes everything in range visible). Scoring: IOC path hit, executable under a temp path, hidden component, setuid/setgid bit.

11. Events tab v0.6

The "what happened around <time>" view: one time-sorted, scored axis merging logons, package activity, boots, journal gaps (and journal contents once loaded via WSL), audit/AppArmor denials, notable log events, and timestamped shell history. This is usually the fastest way to reconstruct an intrusion window.

  • Filters: free-text search, from/to dates, min-score (any / ≥1 / ≥3 / ≥5), and kind (logon, package, boot, journal-gap, sudo, ssh, unit, seccomp, apparmor, acctchg, history, …).
  • Frozen header while scrolling; host-TZ toggle; Export → CSV writes the full unfiltered set with epochs.

12. Log Integrity tab v0.4–v0.6

systemd journal coverage & gap detection

Journal filenames encode a head timestamp and a generation id, so coverage is mapped without parsing the binary format: archived span, rotation cadence, and flagged gaps - a gap ≥3× the median rotation interval, worst when paired with a generation change (the classic journalctl --vacuum / journal-deletion anti-forensics tell). A Boots table from journalctl --list-boots adds the authoritative boot list with between-boot gap checks.

Journal contents via WSL v0.6

When wsl.exe is installed and the capture carries binary journals, Load journal contents (WSL) runs journalctl -D <dir> -o json against the extracted journal and pulls high-signal lines - sudo, sshd, unit start/stop, seccomp, segfaults, OOM, AppArmor, account changes - into the Events timeline. On demand (button) because cold WSL + large journals can take a minute; requires a WSL journalctl new enough for the journal's format (a clear message tells you if it isn't).

13. IOC hits tab

v0.3 screenshot; unchanged in anatomy.

A tree-wide line sweep of every texty file <5 MB against your IOC/keyword list (case-insensitive substring; results also written to ioc_hits.csv), plus SHA1-shaped terms matched against hash_executables to catch dormant on-disk implants that aren't running. The sweep is capped at 20,000 files and says so if it hits the cap.

14. Inventory tab

v0.3 screenshot; the catalog has since grown to ~55 artifacts.

A present/absent map of every key artifact the tool knows about - processes, network, logons, accounts (incl. sshd_config, pam.d, auditd), persistence (incl. live timers and null-passphrase keys), system, filesystem, logs (incl. journal, sar, cloud-init), containers - each with size and an open folder link. Use it to see at a glance what this capture can and cannot answer. v0.8 adds a Browser databases row fed by the browser-DB sweep - absent on an ir_triage capture is expected (that profile doesn't collect them), not suspicious.

15. More tabs & filters v1.0–v1.6

Sections graduated off the crowded Overview into their own tabs, and several new datasets were added. In tab-bar order:

Apps v0.13

Docker containers (deep-inspected - privileged / host-ns / sensitive mounts / docker diff implants) and Recent package installs from apt/dpkg history with Requested-By attribution. The package table has an Action filter (install / upgrade / remove / purge / …), a User-Requested Only filter, and a green USER-REQ chip on every package a human explicitly asked for (a typed apt install / a Requested-By entry) - separating them at a glance from the wall of unattended-upgrade and cron noise - plus a Suspicious Only filter (dual-use tooling watchlist + IOC matches).

Users v0.11

Every account from passwd with shell, groups, shadow lock state, SSH-key and shell-history counts, last login, and an Enabled verdict (interactive shell AND a usable password or SSH keys; ? when shadow wasn't collected). Multiselect account filter, an enabled-only toggle, and a By group view (every group with GID + members).

SSH Keys v1.1

The key material in the capture: authorized_keys (who can log in, with source-restriction and forced-command flags), per-user identity keys (public/private, passphrase state), and host keys. A short key-id (base64 head…tail) lets you match the same key across accounts by eye; a null-passphrase private key, and an unrestricted or forced-command authorized_keys entry, are flagged.

Persistence v0.13 · sweeps expanded v1.7-v1.8

The scored persistence command lines and the config/persistence inventory (formerly on the Overview). A min-score filter (default ≥1; "all incl. benign" surfaces the hundreds of stock cron/systemd lines) decides how much noise to show. Sources: cron (v1.7 also the cron.{hourly,daily,weekly,monthly} run-parts dirs & anacrontab), systemd units & timers (system + user), authorized_keys, rc.local (v1.7 + the RHEL /etc/rc.d/rc.local and /etc/init.d scripts), ld.so.preload, motd & APT hooks, modprobe, XDG autostart, shell startup files, and PAM module directories.

RAT / C2 sweeps v1.7. The tab now covers the hiding spots current Linux RAT/C2 write-ups flag that the earlier sweeps missed:

  • System-wide login execution - /etc/profile, profile.d/*.sh, /etc/bash.bashrc, /etc/zsh/*, /etc/environment (the per-user rc sweep can't see these).
  • SSH-login hooks - /etc/ssh/sshrc and per-user ~/.ssh/rc, which run on every inbound SSH session before the shell - a stealthy, under-triaged trigger. Tag SSH-RC.
  • Network-event triggers (beacon-on-connectivity, boot-independent) - NetworkManager/dispatcher.d, if-up.d/if-post-up.d/if-down.d, dhclient-exit-hooks.d, ppp/ip-up.d. Tag NET-TRIGGER.
  • Boot / device autostart - /etc/modules & modules-load.d (kernel-module autoload, MOD-AUTOLOAD), udev rules with RUN+=/PROGRAM= (UDEV-RUN), binfmt_misc handlers (BINFMT), at jobs.
  • Injection & loader vectors v1.7.1 - systemd Environment=LD_PRELOAD, EnvironmentFile= from a writable path, a .socket unit's network ListenStream=, systemd generators (/etc/systemd/system-generators), /etc/ld.so.conf.d paths outside the standard lib dirs, and authorized_keys environment=.
  • Structural correlation v1.8 - a persistence entry's executable is resolved and, when it runs from a non-standard path, matched against the running process list and its sockets. A live process holding an established public connection escalates the finding (PERS-C2); a running-but-quiet or flagged process is noted (PERS-RUN/PERS-SUSPROC). This is what catches a renamed implant that scores zero on its command string alone.
  • systemd depth v1.8 - drop-in overrides (<unit>.service.d/*.conf, merged over the primary unit) are now parsed; broader load paths (/run, /usr/local/lib, XDG, ~/.local/share); and .path/.automount Unit= + .socket Service= trigger targets.
  • Auth-stack breadth v1.8 - sshd AuthorizedKeysCommand / AuthorizedPrincipalsCommand/File / TrustedUserCAKeys / PermitUserEnvironment and ~/.ssh/environment; PAM pam_exec and absolute module paths across the account/password/session stacks (not just auth).
  • Boot & activation v1.8 - cloud-init scripts/per-boot|per-instance|per-once and user-data runcmd/bootcmd; D-Bus service-activation Exec=.

Package-shipped scripts (stock cron and network hooks) are whitelisted or listed as low-score leads; the content sweep and the C2 lens below elevate a genuinely malicious one. On top of the locations, every persistence command line gets a RAT/C2 tradecraft lens - see Scoring → Persistence - that flags reverse shells, backpipes, encoded loaders, known-implant names, and hardcoded callback IPs wherever they appear.

Actions v1.2

The full executed-command record from each user's bash/zsh/sh history - every line, not just the hostile ones the Persistence tab keeps - timestamped where the shell recorded times (bash HISTTIMEFORMAT, zsh extended history). Each line is scored across the attacker playbook (see Scoring → Actions): recon, credential access, scanners/offensive tools, download→chmod→exec, account & persistence edits, remote copy/SSH, packet capture, kernel modules, web-shell drops, anti-forensics, defence-evasion, elevation. Per-user filter, case-window and host-TZ aware; click a timestamped row to pivot the Timeline ±15 min.

Containers v1.9

A dedicated tab (left of Apache Web Server; the container table moved off Apps) built strictly from the evidence a UAC capture actually holds under live_response/containers: docker/podman info + version, the ps -a / container ls --all --size listings, per-container inspect, diff and top/stats snapshots, image/network/volume listings, and the LXD tables. Everything on it is point-in-time state - it is never presented as Docker event history, and the tab footer states explicitly what these artifacts can NOT show (docker events, docker exec history, destroyed containers, container stdout/stderr logs, in-container process history, the contents or hashes behind diff paths, volume contents, image provenance, per-namespace network state).

  • Collection coverage first - container evidence varies sharply by profile and permissions, so the tab opens with a per-evidence status map that distinguishes present / partial (N of M containers) / permission denied / command failed (daemon unreachable) / runtime not installed / not collected by this profile. Absence is surfaced, never silently blank.
  • Runtime metadata - versions (engine, containerd, runc), storage/logging/cgroup drivers, root dir, security options, live restore, swarm state, registries/mirrors/proxies and counts, with runtime-level findings: a non-loopback insecure registry, a non-standard docker root dir, missing AppArmor/SELinux/seccomp support, active swarm, an egress proxy, registry mirrors. These also feed a new Security-posture check.
  • Inventory incl. stopped containers - a stopped container may be a failed malware execution or terminated attacker tooling, so exited rows stay visible with their exit code. Click a row for the deep-dive.
  • Inspect deep-dive - lifecycle (created / last started / finished, with exit code, OOM-killed state, restart count/policy and health status; the timestamps click-pivot the Timeline and join the Events axis labelled inspect-derived state), execution configuration (entrypoint, command, user, workdir, image, compose labels, stop signal, health-check command), and the environment with values redacted by default - names always show, and a value that is itself an indicator (LD_PRELOAD, a public IP:port endpoint, a temp path, a URL) always shows; a checkbox reveals the rest. The v1.7 C2/implant lens runs over the entrypoint, command and health check.
  • Isolation & host exposure - privileged, host pid/network/ipc/uts/userns namespaces, docker.sock mounts, sensitive host mounts (/, /etc, /root, /proc, /sys, /dev, /boot - with rw/ro state), dangerous added capabilities (SYS_ADMIN, SYS_MODULE, SYS_PTRACE, DAC_READ_SEARCH, NET_ADMIN, ALL), AppArmor/seccomp unconfined, device mappings, no-new-privileges and read-only-rootfs state, and 0.0.0.0/:: port publishes. Combinations score harder than single settings: root + privileged + docker.sock is called out as critical host-control exposure.
  • Mounts & network - mount table with a find-in-Timeline pivot on each host source path (a recently modified host file mounted into a container is a lead), container IP/MAC/gateway/networks, and published ports cross-checked against the host's ss listeners - labelled corroboration that the publish was live, never proof of which process owned a connection.
  • In-container processes - the docker top snapshot lists in-container processes with host PIDs; a PID present in the collected process list is clickable, and inspect's State.Pid gives the container-to-host-process join: a container whose host process is already flagged on the Processes tab is escalated (CTR-SUSPROC).
  • Filesystem changes (docker diff) - added/changed/deleted counts, a per-top-level-directory roll-up, and scored suspicious paths: credential and PAM files, cron/systemd/init persistence locations, ld.so, SSH keys and config, /usr/local/bin drops, web roots, temp/ram paths, hidden paths, package-manager databases, and log/history deletions (anti-forensics). A suspicious-only filter narrows the table. The caveat is stated in-line: diff carries path and change type only - the changed files' contents, ownership and hashes were not acquired.
  • Persistence nuance - a restart policy of always/unless-stopped alone is routine operations and scores nothing; it only adds +1 (CTR-PERSIST) when the container is already suspicious for other reasons.

Everything exports to a single CSV (coverage, runtime findings, inventory with lifecycle, per-path diff findings), and the HTML report gains a matching Containers section.

Apache Web Server v1.6

Renamed from Apache logs. It now opens with an Apache Configuration summary parsed from the enabled config (/etc/apache2 Debian layout or /etc/httpd RHEL layout): key effective directives - ServerTokens/Signature, TraceEnable, run-as user (${APACHE_RUN_USER} resolved via envvars), Listen, DocumentRoot, TLS, directory listing, mod_status, modules - each with an at-a-glance OK / REVIEW chip. The scored CIS versions of these checks live on the Security tab. Below the config are the access logs (apache2 / nginx / httpd combined format), scored for initial-access hunting: POSTs to script URIs, traversal / encoded payloads, tool user-agents.

Audit v0.11

auditd execve records with hex-decoded arguments and auid attribution, plus USER_AUTH / USER_LOGIN events - the closest Linux gets to process-creation telemetry. Present only when the host ran auditd. A brute-force-only filter narrows to failed auth records from a source that failed ≥5 times.

Non-Business Hours filter v1.6

A checkbox on Logons, Actions, Events, Apache Web Server and Timeline that keeps only rows timestamped outside business hours in the collector host's local time - Monday–Friday 08:00–18:00, with the whole weekend treated as out-of-hours. The host's timezone offset comes from uac.log (a capture that carried none is treated as UTC, and the chip says so); the analyst's own clock is never used. Rows with no timestamp are hidden while it's on; a Timeline row stays if any of its four MAC-times is out-of-hours; the Logons By source IP aggregation is rebuilt from out-of-hours events only; and the filtered report honours it.

Row limits & the "Maximum Rows Exceeded" warning v1.6

Tables render at most 10,000 rows (a render-time performance bound for the HTA), and package / persistence / lateral-movement panels have their own caps. Whenever the filtered set is larger than what's shown, a prominent red ⚠ Maximum Rows Exceeded chip appears with a tooltip - so a truncated grid can never read as the complete picture. The two uncapped escape hatches: narrow the view with the search box and filters, or use Export → CSV, which always writes every filtered row. For the Timeline specifically, the case window (/from /to) narrows at parse time and brings rows beyond the 50k parse cap into range.

18. Case window

20. Methodology

Full ir_triage capture

  1. Overview - read the caveats (profile, non-root, long paths), then Top findings and the ATT&CK chips. Check boot time vs the incident window.
  2. Security - the fastest orientation: read the FAIL/WARN rollup, work the correlations (brute-force success, download→chmod→exec, persistence-after-logon), then the hardening posture. Each finding pivots to its evidence - treat them as leads to confirm on the source tab, not verdicts.
  3. Persistence - anything scored is the attacker's foothold candidate list. The When column ties history entries into the story.
  4. Actions - what the operator actually typed, scored across the playbook; sort by score, then read chronologically around your pivot times.
  5. Processes - flagged rows first; use Tree view to see lineage; open detail for ancestry, sockets, ATT&CK. A deleted binary or LD_PRELOAD is rarely innocent.
  6. Network - external ESTABLISHED and unexplained LISTEN; trust the hidden-socket flags.
  7. Logons - filter Result=accepted, then failed clusters; follow the raw-file links to verify anything load-bearing.
  8. Events - set the case window and read the sequence around your pivot times; load journal contents (WSL) if coverage matters.
  9. Logs - integrity findings (gaps = anti-forensics), service inventory for exposure, cloud-init for provisioning-time persistence.
  10. IOC sweep - drop in known indicators (hashes, IPs, filenames) and re-Analyze; SHA1s also match dormant on-disk executables.

Logs-only capture (-a files/logs/*)

Processes/Network/Timeline are empty by design - the signal lives in Logons (auth + wtmp/btmp/lastlog binaries + rotated history), Events, and Logs (journal coverage/gaps, service inventory as your process-list substitute, sar for CPU anomalies, integrity checks).

Non-root capture

Expect: no shadow/sudoers, no other-user /proc detail, no btmp, permission-denied caveats. The tool recovers socket ownership from /proc/net and keeps everything else honest - treat "absent" artifacts as uncollected, not clean.

21. Notes & limitations

  • tar.exe exits 1 on UAC tarballs (Unix symlinks) - expected; success = uac.log exists.
  • Long paths: files beyond 260 chars extract but their contents are unreadable to the HTA layer; the app warns pre-extraction, audits and reports the exact count, and journal coverage is immune. Fix: shorter output directory.
  • WSL journal ingestion needs wsl.exe plus a journalctl at least as new as the journal format; it's a button, not automatic, and caps its pull (20k lines).
  • Times are UTC by default; the host TZ toggle (Logons/Timeline/Events/Logs) re-renders using the collector's offset from uac.log. Classic syslog lines carry no year - the capture year is stamped; last -a -F and ISO logs are exact.
  • Caps, all reported when hit: timeline 50k stored rows; IOC sweep 20k files, 5 MB/file, 5k hits; journal pull 20k lines; displayed rows 10k per table (packages/persistence/lateral have their own). Any capped table shows a red ⚠ Maximum Rows Exceeded chip - narrow with filters or Export → CSV, which always writes the full set.
  • Scoring is heuristic. It ranks where to look first; it is not a verdict. Desktop-Linux noise (GNOME memfds, X11 lock files, upgraded libraries, renamed threads, Debian's trailing pam_permit) is explicitly tuned out, but every environment has its own benign weirdness - read the reasons, follow the raw files.
  • Collector commands that hung/timed out on the source host show as failed on the Overview and leave empty files - normal on hosts with broken tools.
  • Browser databases v0.8 are only present in full-profile (or -a files/browsers/*) captures - ir_triage skips them, so the hand-off section staying hidden is normal. The SQLECmd hand-off launches the wrapper and prefills it; it never parses SQLite itself, and per-file hand-off of a >260-char path is blocked (use Analyze all - SQLECmd's own directory sweep is long-path capable).

22. Conventions

  • Row tint: red = score ≥5, amber = 3–4; red left border = IOC hit. IOC tags are red, heuristic tags amber, informational grey.
  • Verdict vocabulary (Security tab): FAIL = confirmed-ish indicator/weakness, WARN = worth a look, PASS = checked and clean, N-A = the source artifact wasn't collected (never a false all-clear).
  • ⚠ Maximum Rows Exceeded means the filtered set is larger than the render cap - the grid is truncated; filter further or Export → CSV for everything.
  • Click a row for the detail pane (reasons, ATT&CK, copy); click column headers to sort; drag header edges to resize (double-click to reset; widths persist per view).
  • open folder links select the raw file in Explorer - the extracted capture is the evidence; the app is a lens.
  • Self-update: the Update button appears when a newer release exists on GitHub and replaces the .hta in place; on success v0.9.1 the app closes automatically after you click OK - just reopen it to run the new version.
